# devices-pcie-bcm2711 — upstream-readiness review

- **Area:** BCM2711 PCIe root-complex bring-up + outbound/inbound window programming + VL805 firmware-reset mailbox (in-process, owned by the `usb` daemon's xhci PHY init).
- **Files:** `usb/xhci/bcm2711-pcie.c` (1555 lines, new), `usb/xhci/bcm2711-pcie.h` (new), `usb/xhci/phy-aarch64a72-generic.c` (new), `usb/xhci/Makefile` (new).
- **Repo:** `phoenix-rtos-devices`  base `d511e0f` (origin/master) → head `ebac8e4` (master).
- **Referents used:** `pcie/server/pcie.c` (the project's standalone PCIe daemon — same config-space/scan plumbing this file was forked from; itself a separate review area, also project-authored on top of an upstream that has no BCM2711 logic), `usb/xhci/xhci.c` header for the no-op stub convention, `usb/ehci/ehci.c` for the `va2pa` idiom.

## Verdict

The driver **logic is largely correct** — I verified the FIX-12 window/BAR-size arithmetic, the MMIO ordering, and the 64-bit register writes by hand and found no live bug in them. The upstream-readiness blocker is **presentation**: the file reads as a debugging journal (FIX-1 through FIX-18 archaeology, an empirical sweep table, disproven-hypothesis narration, and ~12 `snprintf`+`debug()` register-dump blocks), plus one **dead code path** (`USB_HCD_PCIE_DRIVE_ONLY`) that nothing in the tree sets. Most findings are ROLLBACK / COMMENT with a couple STYLE/ARCH; **zero true high-sev BUGs**.

## Findings (ordered by severity)

### 1. `bcm2711-pcie.c:1368-1427` · ROLLBACK · sev=high
**WHAT:** The entire `USB_HCD_PCIE_DRIVE_ONLY` early-out path (a ~60-line block that mmaps the host bridge, polls MISC_STATUS for link, and returns EOK/-ENODEV without doing bring-up) is dead code. `grep -rn USB_HCD_PCIE_DRIVE_ONLY` across `phoenix-rtos-project`, `-lwip`, and `-devices` shows the string is **only read here** (lines 1370 comment, 1384 `getenv`) and **set nowhere**. **WHY:** Per MEMORY/`project_usb_in_lwip_process` the USB stack was an in-lwip PoC that set this env; USB is now a standalone daemon, so the branch is never taken. Carrying a dead, hardware-specific control-flow branch with a 16-line comment narrating a retired "drive-only PoC" is exactly the kind of thing the maintainers will flag. **REC:** Delete lines 1368-1427 (the whole `#ifdef`-guarded `if (getenv(...))` block); the `ret = pcie_cfgInitBcm2711(&cfgio);` at 1428 becomes the unconditional first statement under the `#ifdef`. Confirm no `setenv("USB_HCD_PCIE_DRIVE_ONLY")` lands in a later integration before removing. **NEEDS-HW** (removes a control-flow branch; gate on build + netboot boot-to-psh smoke — should be a no-op since the branch is never entered, but it touches init flow).

### 2. `bcm2711-pcie.c:701-718, 756-808, 1042-1114, 1182-1223, 1244-1252` · ROLLBACK · sev=med
**WHAT:** ~12 diagnostic `snprintf(dbgbuf,...)` + `debug(dbgbuf)` register-dump blocks (RC_BAR2 readback, "RC PCIe Cap @0xAC DCAP/DCTL", VL805 DCAP/DCTL/LNKCAP/LNKCTL pre/post, "pre-mailbox CMD", "final CMD", BAR0 readback). None carry a `TODO(TD-xx)` marker. These are FIX-8/9/10/11 instrumentation left in after the hypotheses resolved. **WHY:** Per the review's ROLLBACK category and `feedback_build_validate_cleanups`, diagnostic-only prints whose hypothesis is settled must go before presentation; they also bloat a hot boot path. **CRITICAL NUANCE:** several blocks interleave a pure print with a *functional* register write that reuses the read value — e.g. 1073-1114 reads `devctl` once and uses it for BOTH the debug print AND the NO_SNOOP-clear / sticky-clear / ASPM-disable writes (USB-FIX-9/10/11). You **cannot** blind-delete those blocks. **REC:** Strip only the `char dbgbuf[..]` / `snprintf` / `debug()` statements and the readback-only locals (e.g. `lo_rb`,`hi_rb`,`devctl_post`,`lnkctl_post`,`bar_lo`/`bar_hi` *when used only for a print*). KEEP the `fprintf(stderr, ...)` *error* reports (BAR0-mismatch 1178, link-down 588, fw-not-up 1148) — those are legitimate failure diagnostics. The pure print-removal is **APPLY-SAFE**; where a print shares a read with a functional write (NO_SNOOP/ASPM/sticky/BAR), that write is **NEEDS-HW** — leave the write, remove only the print around it.

### 3. `bcm2711-pcie.c:1442-1485` (sweep table) + 1006-1036, 1194-1223, 1204-1211 · COMMENT · sev=med
**WHAT:** Long disproven-hypothesis / lab-journal comments: the empirical settle-time sweep table (1469-1483, "wait | rc=-19 | rc=-110 … bridge drifted"), "mode B in our 10-cycle experiment" (1024), "the symptom we've been chasing all day" (356, 522 region), the keepalive archaeology (1182-1223: "Historically this was a keepalive against a bridge-translation invalidation bug … that turned out to be a side effect … and is fixed by the dev-0-only sweep"), and the negative-result nGnRnE note (1204-1211: "Tested as a candidate … it is NOT the fix, but the alignment is kept"). **WHY:** README COMMENT category — stale/excessive/noise comments and disproven-hypothesis narration. A maintainer reading this for the first time needs the *current* rationale (why 50 ms, why nGnRnE, why dev-0-only), not the debugging history of how it was found. **REC:** Replace each with a 1-2 line statement of the present invariant, e.g. settle: "50 ms post-reprogram settle: empirically the bridge translation needs time to propagate; longer windows let start4.elf re-touch PCIe and invalidate it." Drop "all day", "10-cycle experiment", "Historically… turned out to be". **APPLY-SAFE** (comment-only).

### 4. `bcm2711-pcie.c` whole file vs `pcie/server/pcie.c` · ARCH · sev=med
**WHAT:** This file is a near-verbatim fork of `pcie/server/pcie.c`: `pcie_cfgio_t`, `pcie_cfgRead8/16`, `ecamReg*`, `print_bars`, `print_capabilities`, `scanFunc`, `pcie_scanBus`, the BCM2711 indexed-cfg block, the mailbox notify, and the `UPPER/LOWER_32_BITS`/`readReg`/`writeReg`/`writeRegMsk` helpers are duplicated almost line-for-line. **Referent:** `pcie/server/pcie.c` (the standalone PCIe daemon) contains the same functions. **WHY:** Two copies of the PCIe scan + BCM2711 bring-up will drift (already diverging: this copy added the non-page-aligned mailbox mmap fixup and the `USB_HCD_PCIE_DRIVE_ONLY` path, the daemon copy did not). Phoenix convention is shared bus code factored into a lib, not copy-pasted per consumer. **REC:** Before upstreaming, decide the single owner of BCM2711 PCIe bring-up. The merged design (this file, owned by the usb daemon) is the documented direction per the Makefile comment and `bcm2711_pcie_initVL805`'s header — so the standalone `pcie/server` copy of the BCM2711 path should be retired, or the common scan/cfg plumbing hoisted to a shared `pcie/common` lib both include. Cross-reference the `devices-pcie-server` review. **NEEDS-HW** (build-system / ownership change).

### 5. `bcm2711-pcie.c:6-7` vs `bcm2711-pcie.h:6` · STYLE · sev=low
**WHAT:** The `.c` header reads `Copyright 2025 Phoenix Systems / Author: Dariusz Sabala` and titles itself "PCI Express driver server" — both carried verbatim from the forked `pcie/server/pcie.c`. The `.h` (and the sibling new files `phy-aarch64a72-generic.c`, `xhci.c`) read `Copyright 2026 Phoenix Systems` with no author and a BCM2711-specific title. **WHY:** Inconsistent year/author/title within the same new module; the `.c` title misdescribes it (it is not the "driver server" — that's `pcie/server`). **Referent:** `usb/xhci/xhci.c:6` ("Copyright 2026 Phoenix Systems", no author line) and `bcm2711-pcie.h:1-9`. **REC:** Make the `.c` header match: `Copyright 2026 Phoenix Systems`, title "BCM2711 PCIe bridge bring-up for VL805 USB (xhci PHY)" to match the `.h`; keep an author line only if the project convention does (xhci.c omits it). **APPLY-SAFE** (comment/header only).

### 6. `bcm2711-pcie.c:295-301` · BUG (latent) · sev=low
**WHAT:** `msgaddr = va2pa(msgbuf)` returns `addr_t` (64-bit, per `libphoenix/include/sys/mman.h:47`); `msg = ((uint32_t)msgaddr & ~0xfu) | RPI_MBOX_PROP_CHANNEL` truncates it to 32 bits before handing it to the VideoCore property mailbox. **WHY:** Correct only under the unstated invariant that the `MAP_CONTIGUOUS` allocation lands below 4 GiB (VC property-mailbox bus addresses are 32-bit). On a 4 GiB-DRAM Pi 4 with the TD-15 high-memory unlock, a contiguous buffer *could* be placed above 4 GiB, in which case the truncated address points the GPU at the wrong PA and the notify silently corrupts memory / returns garbage. There is a `va2pa == -1` guard (296) but none on the high bits. The same pattern exists in the `pcie/server/pcie.c` referent, so it is inherited, not newly introduced. **REC:** Add an explicit check `if (msgaddr > 0xfffffff0u) return -EFAULT;` (or assert the allocation is 32-bit), and a one-line comment stating the 32-bit-bus-address invariant. **NEEDS-HW** (touches the mailbox handshake; document, do not blind-apply).

### 7. `bcm2711-pcie.c:114-115, 125` (file-static mutable state) · ARCH · sev=low
**WHAT:** Two file-static singletons hold cross-call state: `bcm2711_pcie_xhciMmio` (the pre-created MMIO mapping) and `bcm2711_pcie_lastCtx` (the bridge context, leaked-by-design so `resettleOutboundWindow` can replay writes). **WHY:** Phoenix HCD drivers thread per-controller state through an explicit context struct rather than file-statics (referent: `usb/ehci/ehci.c` passes `ehci_t *` everywhere; `xhci.c` carries `xhci_t *xhci`). The singletons are defensible here because there is exactly one BCM2711 bridge and the merged-process lifetime is the daemon's, but they make the module non-reentrant and the leak intentional-but-implicit. **REC:** Acceptable for a single-instance board; if upstreamed, note in the header comment that the module is explicitly single-instance and the bridge mapping is intentionally held for daemon lifetime (the `initVL805` bottom comment already explains the leak — just reference it from the static decls, which the FIX-already does at 118-124). Lowest priority. **NEEDS-HW** (refactor) — recommend document-only.

## Verified-correct (explicitly checked, NOT findings)

- **FIX-12 BAR2-size encoding (`bcm2711EncodeBar2Size`, 594-636):** counts trailing zeros to get log2(size), range-guards `15..46`, returns `shift-15`. For 4 GiB → 17. No off-by-20 regression; the historical `shift=20` loop bug is genuinely gone.
- **FIX-12b BAR2-LO mask (681-682):** `value &= ~0xfffff000u; value |= LOWER_32_BITS(pcieAddr) & 0xfffff000u` no longer clears bit 4 of the 5-bit size field. Correct.
- **Outbound window arithmetic (`bcm2711SetOutboundWindow0`, 639-664):** for cpu=0x600000000, size=64 MiB → baseMb=0x6000, limitMb=0x603F; low/HI split across BASE_LIMIT (bits 4-15 base / 20-31 limit) and BASE_HI/LIMIT_HI (bits >12 of the MB value) reconstructs correctly.
- **MMIO ordering:** bridge regs are `MAP_DEVICE` (Device-nGnRE, non-reordering) so back-to-back register writes need no inter-write barrier; the `dsb sy; isb` before the handoff (1494-1495) and at the end of `resettleOutboundWindow` (1549-1550) are correct and sufficient.
- **64-bit BAR0 write (1168-1171):** low word masked `& 0xfffffff0u` (preserves type bits cleared), high word `>> 32`. Correct.
- **`PCI_COMMAND` 32-bit write zeroing PCI_STATUS (0x06):** benign — STATUS is RW1C/RO, inherited from the referent. Not a finding.
