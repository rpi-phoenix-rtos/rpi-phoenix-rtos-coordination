# Task #56 — X11 named bitmap fonts on Xphoenix (kdrive)

Status: **ROOT-CAUSED + FIXED (host-side); awaiting HW re-validation.**
Date: 2026-06-29. Host-side only (no Pi boot — main session owns the UART).

## ROOT CAUSE (confirmed on HW + in source) — libphoenix fdopen rejects "rt"

The HW `xfontprobe` run returned `XGetFontPath -> 1 element "built-ins"`,
`disk=0/3`. The disk font directory was **never in the running font path** — the
disk FPE was silently dropped at init. Traced end-to-end:

1. `os/utils.c:791-793` — `-fp /usr/share/fonts/X11/misc` sets
   `defaultFontPath = "/usr/share/fonts/X11/misc"`.
2. `dix/main.c:225` — `SetDefaultFontPath(defaultFontPath)` which only ever
   **appends** `,built-ins` (`dixfonts.c:1714-1715`) — it never replaces. So the
   intended path is `"/usr/share/fonts/X11/misc,built-ins"`. (Coordinator hypothesis
   (a) "built-ins replaces -fp" is REFUTED.)
3. `dixfonts.c:1637` — `SetFontPathElements` calls `init_fpe` on each element. The
   misc element's `init_fpe` = libXfont2 `FontFileInitFPE` ->
   `FontFileReadDirectory("/usr/share/fonts/X11/misc")`.
4. `libXfont2 dirfile.c:99-101` —
   `file_fd = open(dir_file, O_RDONLY|O_NOFOLLOW); file = fdopen(file_fd, "rt");`
5. **libphoenix `stdio/file.c` `string2mode("rt")` returns -1** (file.c:66-83): the
   'r' branch sees the next char 't' (not 'b'/'\0'/'+'/'c') and falls to
   `return -1`. So `fdopen` sets `errno = EINVAL` and returns NULL (file.c:227-230).
6. Back in `dirfile.c`: `if (file)` is false, `else if (errno != ENOENT)`
   (EINVAL != ENOENT) -> returns **BadFontPath** (dirfile.c:153-154).
7. `FontFileInitFPE` returns the error; `SetFontPathElements` drops the element and
   logs via **`DebugF`** (`dixfonts.c:1640-1645`) — a **no-op without -DDEBUG**,
   which is why no "Could not init font path element" ever appeared in the logs.
   Only `built-ins` survives. (Coordinator hypothesis (b) is CORRECT.)

This is deterministic and **compression/content-independent** — it fails before any
`.pcf` is opened (the `.pcf` read path, `fileio.c FontFileOpen`, uses raw `open()` +
`BufFileOpenRead`, NOT fdopen). That is exactly why uncompressing the fonts never
helped.

## THE FIX (host-side, shipped this session)

`patches/libXfont2-2.0.6-phoenix-fdopen-rt.patch`: change `fdopen(fd, "rt")` ->
`fdopen(fd, "r")` at `dirfile.c:101` (fonts.dir) and `:298` (fonts.alias). POSIX
fopen/fdopen ignore the 't' flag and "r" is text mode by default, so "r" is
equivalent on every conformant libc AND is accepted by libphoenix. The
`"%d\n"` / `"%255[^\n]\n"` fscanf reads on these streams are handled conformantly by
libphoenix `stdio/scanf.c` (CT_CCL scanset + integer paths are standard
BSD-derived), so this is a **complete** fix, not a half-fix.

Applied to the build tree, rebuilt libXfont2.a, **relinked Xphoenix** (0 undefined
symbols), published to `artifacts/x11/Xphoenix` and staged to
`/srv/phoenix-rpi4-nfs/bin/Xphoenix`. Verified the relink used the freshly rebuilt
archive (lib mtime precedes the server link).

The build script applies this patch automatically on a fresh build:
`build-x11-phoenix.sh` `xbuild` -> `apply_patches libXfont2-2.0.6` globs
`patches/libXfont2-2.0.6*.patch` (`patch -p1 -N`, idempotent).

Follow-up (NOT done here — wider blast radius): make libphoenix `string2mode()`
ignore a 't' flag so any port using `fdopen(..,"rt")` works. That is a `--scope
core` change; the localized X-side patch unblocks #56 now without touching libc.

## TL;DR (earlier investigation — the three briefed hypotheses)

The three hypotheses in the task briefing are **all refuted by the currently
staged artifacts** (server dated 2026-06-28 02:33, fonts staged 2026-06-28 22:58):

1. "PCF/bitmap disk FPE not registered" — **FALSE.** `FontFileRegisterFpeFunctions`,
   `BitmapRegisterFontFileFunctions`, `pcfReadFont`/`pcfReadFontInfo` are all
   present (defined `T`) in `artifacts/x11/Xphoenix`.
2. "libXfont2 built `--without-zlib`, so `.pcf.gz` can't be decompressed" — **FALSE
   for the staged build.** `gunzip.o` IS in `libXfont2.a`; it references
   `inflate`/`inflateInit2_`/`inflateEnd`; `libz.a` provides them; the server links
   `-lz`; `BufFilePushZIP` + `inflate` are defined `T` in the server with **0
   undefined symbols**. The `--without-zlib` in `build-x11-phoenix.sh:170` is on
   **freetype**, not libXfont2 — libXfont2 auto-detected the zlib at
   `/tmp/x11-phoenix/include`. **Rebuilding libXfont2 "with zlib" is a no-op.**
3. "font path wrong/empty" — **FALSE.** Compiled `COMPILEDDEFAULTFONTPATH` includes
   `/usr/share/fonts/X11/misc/`, and the launcher explicitly passes
   `-fp /usr/share/fonts/X11/misc` (confirmed in the live UART log, see below).

So the font *machinery* is present and the *data* is correctly staged. The failure
is a **runtime behaviour**, and the failing **layer is not yet confirmed** — it is
either (A) the server's per-font disk load path, or (B) a **client-side double-free**
in xcalc. A minimal libX11-only probe (`xfontprobe`, built + staged this session)
discriminates the two in one HW cycle.

`tools/x11-port/src/libXfont2-2.0.6/src/builtins/fonts.c`:

**Attribution caveat (do not over-read a PASS):** `fixed`, `6x13`, `cursor` are ALSO
present on disk (they are standard misc fonts / aliases — see `fonts.alias`). So a
`XLoadQueryFont("6x13")` PASS does **not** tell you which FPE served it (the path
search may have hit either). These three are therefore **controls present in BOTH
back-ends**, not built-in-only probes. The decision rides entirely on the
**disk-only** fonts — `10x20` and the direct `c-80` XLFD (neither is in the built-in
`{6x13,cursor}` set) — plus the `XGetFontPath` output. The probe's VERDICT logic
keys off exactly those (`disk_ok`), so it is robust regardless of this caveat.

**Self-validating closure:** if `fixed` loads (server started) but `10x20` / the
direct XLFD FAIL, that single result *by itself* proves both "built-ins are
auto-active" AND "disk load is broken" — no separate experiment needed. Conversely a
`10x20` PASS proves disk serving works and re-points #56 at the xcalc double-free.

### Data correctly staged (host-side)

- `fonts.dir` header `409`; `8x13.pcf.gz`, `10x20.pcf.gz`, `8x13-ISO8859-1.pcf.gz`

### open() flags are benign on Phoenix

`dirfile.c`/`fileio.c` use `O_RDONLY | O_NOFOLLOW` (+`O_BINARY`,`O_CLOEXEC`).
Phoenix `<phoenix/posix-fcntl.h>` does **not** define `O_NOFOLLOW`/`O_BINARY`, but
`build-x11-phoenix.sh:177` compiles libXfont2 with `-DO_NOFOLLOW=0`, so they reduce
to plain `O_RDONLY`. (The archive built clean, confirming the macros were supplied.)
Not the cause.

### The live failure log (server 2026-06-28)

`artifacts/rpi4b-uart/rpi4b-uart-20260628-230421-xcalc-pcf.log`, lines 405–424:

```
405 startx xcalc
407 xlaunch: starting server: /bin/Xphoenix :0 -ac -nolisten tcp -fp /usr/share/fonts/X11/misc
409 xlaunch: server socket present after ~30 ms
410 xlaunch: starting client[0]: /bin/xcalc (DISPLAY=:0)
411 [fbdev] /dev/fb0: 1920x1080 ...
417 Warning: Cannot convert string "calculator" to type Pixmap
418 Double free detected
419 Warning: Cannot convert string "8x13" to type FontStruct
420 xlaunch: client[0] exited (status=0x46)
```

Two crucial reads of THIS log:
- **No `Could not init font path element ... removing from list!`** — the dix message
  printed when an FPE fails to initialise. Its absence => `fonts.dir` was read OK at
  FPE init. (The disk FPE reads `fonts.dir` for *names* at init; it does not `open()`
  any `.pcf` until a font is actually requested.) So the directory-read path is NOT
  the suspect; if the cause is server-side it is the **lazy per-font load**
  (`FontFileOpenBitmap` -> `open(*.pcf.gz)` -> `BufFilePushZIP` -> `pcfReadFont`).
- `Warning: Cannot convert string "8x13" to type FontStruct` and `... "calculator"
  to type Pixmap` are **libXt client-side** converter warnings (printed by the xcalc
  process), and in stock Xt a failed String->FontStruct conversion is **non-fatal**
  (Xt falls back to a default font). `Double free detected` (line 418) is from
  **libphoenix `stdlib/malloc_dl.c`** — i.e. the **xcalc process double-freed**. The
  abort-like `status=0x46` is consistent with the double-free, NOT with a benign font
  fallback. So the headline "named fonts don't load" may be a *symptom* of an xcalc
  client-side double-free during Xaw init, masking whatever the font layer really does.

### How to read the verdict
- `builtin=3/3 disk=0/3` -> **server per-font disk load broken**; next step =
  instrument `FontFileOpenBitmap`/`pcfReadFont` return paths (server-side), NOT the
  directory read.
- `disk=3/3` -> **font serving works**; the real #56 bug is the **xcalc/Xaw
  client-side double-free** — refocus there (it is a client bug, not a font bug).
- `8x13` FAIL but direct XLFD/`10x20` PASS (or vice-versa) -> narrows to alias
  resolution / `fonts.alias` parsing.

## Host-side fixes applied this session (in `stage-x11-runtime.sh`)

1. **Latent fonts.dir clobber removed.** The old stage script *wrote* a hand-rolled
   3-line `fonts.dir` (only `6x13`+`cursor`) and a 10-line `fonts.alias`. The
   currently-good export has the full 409-entry upstream `fonts.dir` (copied from the
   host misc tree out-of-band). Re-running the OLD script would have silently
   clobbered it back to 3 entries → every named disk font (`8x13`/`9x15`/`10x20`/…)
   would stop resolving. The script now copies the **full** host
   `/usr/share/fonts/X11/misc/.` verbatim (fonts + upstream fonts.dir + fonts.alias),
   which preserves the #51 iso8859-1 fontset chain AND keeps all named fonts.
2. **Encodings DB now staged.** `/usr/share/fonts/X11/encodings/` (incl.
   `encodings.dir`) was **absent** from the export, yet the server references
   `.../encodings/encodings.dir` (libfontenc). Now copied from the host. This is a
   legitimate correctness fix and a **prime suspect** for a server-side per-font
   failure — but it is NOT asserted as the root cause (iso8859-1/iso10646-1 PCFs
   carry their encoding inline, so a missing encodings DB more often degrades glyph
   mapping than NULLs the FontStruct). The probe run discriminates.

Re-ran `stage-x11-runtime.sh`: misc dir still 409 entries; encodings DB now present
